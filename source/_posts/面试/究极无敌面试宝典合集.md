# 2. 数据库

## 2.1 索引

> 帮助数据库高效获取数据的数据结构

**优点：**

索引本身存储的方式具有顺序性，所以进行排序或者等值操作时，数据库可以通过索引快速定位数据而不需要扫描全表，另外，索引通常还具有唯一性约束，有利于维护数据实体完整性。

**缺点：**

1. 索引本身是数据结构，所以需要一定的存储空间开销，随着表规模增大而增大
2. 增删数据时，索引的结构会发生改变，需要一定性能开销
3. 同一类型的索引不适用于所有查询，比如哈希索引不支持范围查询和排序
4. 过多的索引会导致冗余，维护起来比较复杂

**B+ 树：**每一个索引在 InnoDB 里对应一棵B+树

**为什么需要B+树结构？**

当数据量越来越大的时候，二叉树的节点越来越多，如果这些节点存储在外存储器中的话，每访问一个节点，相当于进行了一次I/O操作，那么当从根节点搜索的时候，影响查询效率，所以设计了B树，用于减少树的深度。在B树的基础上又进行改进得到了B+树，其相对来说有以下几个优势：

1. **磁盘读写代价更低**

   B+树的内部结点并没有存储具体信息，只是存储叶结点指针，因此其内部结点相对B树更小。

   如果把所有同一内部结点的关键字存放在同一盘块中，那么盘块所能容纳的关键字数量也越多，一次性读入内存中的需要查找的关键字也就越多，相对来说I/O读写次数也就降低了；

2. **查询效率更加稳定**

   由于非叶结点并不是最终指向文件内容的结点，而只是叶子结点中关键字的指针。所以任何关键字的查找必须走一条从根结点到叶子结点的路。所有关键字查询的路径长度相同，导致每一个数据的查询效率相当。

3. **便于范围查询【最重要原因】**

   B+树只需要去遍历叶子节点就可以实现整棵树的顺序遍历【叶子节点是循环双向链表】。而且在数据库中基于范围的查询是非常频繁的，而B树不支持这样的操作或者说效率太低

   > 从根节点起步的搜索是随机搜索，叶子节点遍历是顺序搜索

**回表：**先搜索二级索引树，再回到主键索引树搜索的过程称为回表。【回表尽量避免，但是若添加普通索引的收益大于回表的开销也能接受】

> 当一个表中同时存在普通索引和主键索引，且此时将普通索引作为查询条件且查询得到的信息不止主键与普通索引时就会造成回表（如id，name为索引，查询gender字段，以name作为查询条件）【定位到普通索引对应的主键，再通过主键去定位到行数据】（每个非主键索引的叶子节点上都是主键的值，主键长度越小，普通索引的叶子节点就越小，普通索引占用的空间也就越小）

**最左前缀原则**：当有联合索引(a,b,c)时，优先从左匹配索引，遇到范围查询则停止使用索引。

> 如a=1 and b>1 and c=1，这种情况下c的索引无法发挥作用，若希望其发挥作用则需要调换条件顺序（优先考虑）或者为c单独建立索引；
>
> 如果建立了(a,b)联合索引，一般就不需要单独在a上建立索引了。但如果要单独查b，就不得不同时维护(a,b)、(b)这两个索引。

|        | 联合索引                     | 唯一索引                         | 主键索引           |
| ------ | ---------------------------- | -------------------------------- | ------------------ |
| 唯一性 | :x:                          | :white_check_mark:（可以有NULL） | :white_check_mark: |
| 用途   | 优化多列组合查询             | 维护数据完整性                   | 唯一标识           |
| 性能   | 查询联合索引一部分时性能下滑 | 相对主键少了唯一性检查           | 增删改较慢         |

### 索引创建原则

1. 数据量大，查询比较频繁的表
2. 区分度大，常作为查询条件，分组，排序的字段
4. 尽量使用联合索引

## 2.2 事务

### ACID特性

+ **原子性（atomicity）**：指事务是一个不可分割的工作单位，事务中的操作要么都发生，要么都不发生

+ **一致性（consistance）**：事务必须使数据库从一个一致性状态变换到另外一个一致性状态

  > 逻辑正确性，比如A转账给B业务中，A金额少100，B金额就会多100，总额不变

+ **隔离性（isolation）**：确保事务的操作对其他并发事务不可见，直到该事务提交

+ **持久性（durability）**：指一个事务一旦被提交，它对数据库中数据的改变就是永久性的

### 隔离等级

> 隔离级别越高，数据一致性越好，但可能会影响并发性能。

+ **读未提交（Read uncommitted）**：最低的隔离级别，一个事务可以读取到另一个事务未提交的数据变更，存在脏读问题。

  > 不使用锁

+ **读已提交（Read committed）**：一个事务只能读取到另一个事务已经提交的数据变更，可能存在不可重复和幻读问题。

  > 每次读取数据时，都获取最新的提交版本。读操作使用共享锁，写操作使用排他锁。【读操作结束共享锁就释放，排他锁直至事务结束才释放】

+ **可重复读（Repeatable read）【默认】**：一个事务在执行期间看到的数据（当下已存在的数据）保持一致，可能存在幻读问题

  > 通过多版本并发控制MVCC实现【查询使用快照读，插入时使用当前读】，在事务开始时创建快照，每次读取都读取快照中的数据，避免不可重复读问题。写操作使用排他锁，防止其他事务修改读取的数据

+ **序列化（Serializable）**：最高隔离级别，在此等级下事务看起来是顺序执行的，通过对相关数据加锁实现，可以解决所有并发问题

  > 使用范围锁或强制两阶段锁协议【扩展阶段（事务获得锁，不释放锁）与收缩阶段（事务释放锁且不再获得新锁）】，确保事务按顺序执行，防止所有并发问题

各并发问题含义👇

+ **脏读**：当一个事务**读取到另一个事务尚未提交的修改时**，产生脏读
+ **不可重复读**：同一查询在同一事务中多次进行，由于**其他提交事务所做的更新，每次返回不同的结果集**
+ **幻读**：同一查询在同一事务中多次发生，由于**其他提交事务所做的插入删除操作，每次返回不同的结果集**

## 2.3 慢查询

> 一般在数据量较大的SQL查询中，如聚合查询和多表查询，若此时相关接口响应时间过长，超过预期时称之为慢查询

### 如何定位？

1. 运维工具SkyWalking监听接口

2. MYSQL自带慢查询日志

   > 日志文件存储在：/var/lib/mysql/localhost-slow.log

   ```mysql

   slow_query_log=1
   # 设置慢日志的时间为2秒，SQL语句执行时间超过2秒，就会视为慢查询，记录慢查询日志
   long query time=2
   ```

### 如何分析原因？

在查询语句前加explain/desc

## 2.4 SQL优化经验

+ 表的设计优化（参考阿里开发手册）

  1. 根据需要设置合适数值范围（tinyint，int，bigint）
  2. 比如设置合适的字符串类型，char定长效率高，varchar可变长度，效率偏低

+ SQL语句优化

  1. 避免使用 select *
  2. SQL语句避免造成索引失效的写法
  3. 避免在 where 中对字段进行表达式操作（索引容易失效、全表计算）
  4. 连接操作时小表驱动大表
  
+ 主从复制，读写分离

  如果数据库的使用场景读的操作比较多的时候，为了避免写的操作所造成的性能影响可以采用读写分离的架
  构。

## 2.5 undo log 和 redo log

+ redo log：记录的是数据页（InnoDB存储引擎磁盘管理的最小单元）的物理修改，服务器宕机可用来同步数据

  > 数据库先访问内存（缓冲池），没有命中再访问磁盘，内存中的数据只有同步到磁盘才算实现了持久化，当脏页由内存调至磁盘出错时就执行 redo log file 保证数据顺利由内存至磁盘

+ undo log：记录的是逻辑日志，当事务回滚时，通过逆操作恢复原来的数据

redo log 保证了事务的持久性；undo log 保证了事务的原子性和一致性；
表锁行锁，MVCC多版本并发控制（查询操作快照读，更新操作当前读）保证隔离性

## 2.6 主从复制原理

核心是BinLog二进制日志，记录DDL（数据定义语言）语句与DML（数据操作语言）语句

1. Master主库在事务提交时，会把数据变更记录在二进制日志文件Binlog中
2. 从库读取主库的二进制日志文件Binlog，写入到从库的中继日志Relay Log
3. slave重做中继日志中的事件，完成数据同步

## 2.7 分库分表

> 主从复制只能分担访问压力而不能解决海量数据存储问题，此时可将数据库数据表进行拆分

1. 水平分库：将一个库的数据拆分到多个库中，解决海量数据存储和高并发的问题

   > 通常需要配合水平分表

2. 水平分表：将一个表的数据拆分为多个表，解决单表存储和性能的问题

3. 垂直分库：根据业务进行拆分，高并发下提高磁盘IO和网络连接数

4. 垂直分表，按照字段使用频繁程度进行拆分，多表互不影响

   > 比如博客description字段存储较大，单独与主键组成一个表

## 2.8 执行一条SQL的请求过程

+ 连接器：建立连接，管理连接、校验用户身份；
+ 查询缓存：查询语句如果命中查询缓存则直接返回，否则继续往下执行。MySQL 8.0 已删除该模块；
+ 解析 SQL：通过解析器对 SQL 查询语句进行词法语法分析，然后构建语法树，用于读取表名、字段等基本信息；
+ 执行 SQL：执行 SQL 共有三个阶段
  + 预处理阶段：检查表或字段是否存在；将 `select *` 中的 `*` 符号扩展为表上的所有列；
  + 优化阶段：优化器生成执行计划；
  + 执行阶段：执行器根据执行计划执行 SQL 语句，调用存储引擎接口进行具体数据操作，并将执行结果返回给客户端；

## 2.9 Redis

### 2.9.1 缓存穿透

> 查询一个不存在的数据，mysql查询不到数据也不会直接写入缓存，导致每次请求都查数据库

**解决方案：布隆过滤器**

布隆过滤器可以理解为以位为单位的超长数组（只存放0和1），用于检索一个元素是否在一个集合中，每个传入的字符都经过不同的哈希函数计算映射到多个位置，将对应位置置1，后续传入值后，只要哈希计算所包含的值全为1，则说明该字符存在（有一定误判率，因为1的位置可能重叠）

因此，Redis预加载数据库数据（预热）时，可将主键放入布隆过滤器，查询发生时，若不通过布隆过滤器，则无需向Redis发起查询

### 2.9.2 缓存击穿

> 给某一个key设置了过期时间，当key过期时恰好有针对此key的大量并发请求，这些请求可能会瞬间把DB压垮（此时DB还来不及将数据同步至于Redis）

**解决方案**

1. 互斥锁

   当线程访问key发现数据不存在时，为此key加互斥锁，其他线程若此时访问相同的key则会被阻塞在此处，接着线程去数据库查询key相关数据并缓存至Redis后，返回访问结果并释放互斥锁，此时其他线程阻塞被唤醒，并顺利命中缓存

2. 逻辑过期

   缓冲数据维护一个过期字段，当过期字段到期后并不在Redis中真正删除，而是同样加互斥锁并开启一个新线程查询数据库并在完成同步后释放锁，不同之处在于此时线程并不会阻塞，而是会返回过期数据

前者保证缓存与数据库的强一致性，但是由于阻塞导致效率偏低；后者不保证数据是实时的，但是效率较高，具体处理方式看业务场景

### 2.9.3 缓存雪崩

> 在同一时段大量的缓存key同时失效或者Redis服务器宕机，导致大量请求到达数据库，带来巨大压力。

**解决方案**

1. 给不同key的过期时间添加随机值，避免同时大量key失效
2. 添加多级缓存或对数据库进行分布式管理

### 2.9.4 mysql与redis数据一致性

1. **对于写操作采用延时双删策略**：删除缓存 ---> 修改数据库 ---> 等待一段时间 ---> 删除缓存

   不论是先删后改还是先改后删，都可能有数据不一致的可能，因此在修改前后都进行缓存删除，保证后续缓存中的数据是正确的，延时的原因在于数据库主从复制需要时间，确认操作顺利执行后再执行删除，因此在等待的这段时间内也可能会出现脏数据的情况

2. **使用读写锁**

   读数据使用共享锁，写数据使用排他锁（写完成后删除缓存数据）

### 2.9.5 redis持久化

+ RDB（Redis Database Backup File, Redis数据备份文件）是一个快照文件，它是把Redis存储的数据写到磁盘上，当Redis宕机恢复数据时，可以通过RDB进行数据恢复。
+ AOF（Append Only File）是追加文件，Redis会将每次写命令存储在此文件中，当Redis宕机后，会将此文件记录的命令执行一遍用于数据恢复。
+ RDB是二进制文件，在保存时体积较小，恢复速度比较快，但是有可能会丢失数据；AOF恢复速度稍慢一些，但是丢失数据的风险小很多，可以通过设置刷盘策略减小数据丢失风险，如每秒检查一次

### 2.9.6 redis数据过期策略

+ 惰性删除：key过期后不主动删除，下次使用时检查发现过期后再进行删除
+ 定期删除：每隔一段时间就对一些key进行检查，删除过期的key

### 2.9.7 redis数据淘汰策略

内存满了有新数据如何处理？

1. 不淘汰任何key，内存满时不允许插入新数据
2. 淘汰有TTL字段的数据中，TTL较短的数据
3. 对全体key/有TTL字段的key，随机淘汰
4. 对全体key/有TTL字段的key，基于LRU算法（最近最少使用）进行淘汰
5. 对全体key/有TTL字段的key，基于LFU算法（最少频率使用）进行淘汰

# 3. 计算机网络

## 3.1 UDP和TCP的区别

1. TCP 是面向连接的，所需资源多，传输速率慢；UDP 是无连接的，所需资源少、传输速率快
2. TCP 的数据传输以**字节流**的形式，UDP 的数据传输是以**报文段**的形式
3. TCP 用检验和、序号、流量控制、拥塞控制、超时重传等机制保证 TCP 连接是可靠的，UDP 则不保证可靠交付，只保证尽力交付
4. TCP 只适用于一对一单播，UDP 可用于广播通信
5. TCP 适用于对可靠性要求较高的场景，比如文件的传输等，UDP 适用于实时性要求较高的场景，比如微信视频通话等

## 3.2 TCP三握手

<img src="HTTPs://z-cloud-pic-1313046262.cos.ap-guangzhou.myqcloud.com/img/202403212255045.png" alt="24" style="zoom:50%;" />

+ `SYN` 为 `1` 的报文段不能携带数据，但会消耗一个**序列号 seq**
+ `ACK=1` 代表这是普通确认报文段，**ack=x+1 表示这是对报文段序列号 seq=x 的确认**

**为什么需要三次握手？**

确保双方间的连接正常建立，如果只有两次握手的话可能会出现一些异常情况，比如：①客户端的 SYN 连接请求发去时间太久，导致了超时重传的发生，但是服务器端接最终收到了该 SYN 报文，如果不经过第三次握手的话服务器端就会错误地开启一个连接；②如果只有两次握手的话，服务器端返回给客户端的确认报文丢失，会导致客户端因为没有收到确认所以关闭了该连接，但服务器端此时已做好了连接准备，造成资源的浪费。

## 2.3 TCP四挥手

<img src="HTTPs://z-cloud-pic-1313046262.cos.ap-guangzhou.myqcloud.com/img/202403212256420.png" alt="25" style="zoom:80%;" />

+ 客户端发起关闭请求，一去一回后进入半关闭状态【客户端不再发送数据，服务端可能还会发】
+ 服务器将自己剩余的数据发送完后也发送一个关闭请求，接着客户端给予回应后服务器关闭，客户机则要等到一段时间后完全关闭**（服务器端没有收到最终确认报文的话，过 1MSL（报文在网络中的最大存活时间）会重新再发送一次 FIN 报文给客户端，如果过了 2MSL 客户端还没有收到新发的 FIN 报文的话，证明服务器端已经收到确认报文并正常关闭连接，此时客户端也可以关闭连接了）**

**为什么需要四次挥手**

因为释放连接时一方收到 FIN 报文但有可能还有数据要继续传输，不能马上释放连接，所以先返回一个确认报文，发送完数据后再断开连接。

## 2.4 TCP如何保证可靠传输

+ 对发送方数据进行编号，接收方收到后会根据校验和进行检查，检查通过返回确认号，发送方接收到确认后才会将数据从缓冲区删除，如果没有收到，则会引起超时重传
+ TCP拥塞控制：通过动态调整发送方的发送速率来适应网络的拥塞程度
  + **慢开始**：初始以较小的拥塞窗口大小开始发送数据，随着确认收到的ACK增加，逐渐增大拥塞窗口
  + **拥塞避免**：当拥塞窗口达到一个阈值时，拥塞避免算法将拥塞窗口大小线性增加而不是指数增加，以减小网络拥塞风险
  + **快重传**：如果发送方连续接收到相同ACK报文（3次），说明有丢包发生，可以快速重传未收到的数据段，避免等待超时时间
  + **快恢复**：当发生快重传时，拥塞窗口进行快恢复，窗口大小减半，接着线性增加窗口大小
+ TCP流量控制
  + **滑动窗口机制**：发送方和接收方各维护一个滑动窗口，用于控制发送和接收数据的速率。发送方根据接收方的窗口大小调整发送数据的量，确保不会发送超过接收方处理能力范围的数据
  + **接收方反馈**：接收方通过发送ACK报文确认已经接收到的数据，同时将窗口大小信息发送给发送方，以指示其可以继续发送的数据量

## 2.5 输入URL到显示网页发生了什么

+ **浏览器解析URL对应的IP地址** 

  DNS 解析（浏览器缓存 -> 本地 DNS -> 根DNS -> 顶级DNS -> 权限DNS）

+ **建立连接，http是TCP三握手，HTTPS在此基础上还有安全认证相关操作**，

+ **浏览器向服务器IP地址发送一个HTTP请求报文** 

  + 应用层按照 HTTP 协议的规定构建一个**HTTP请求报文（请求要干什么）**。应用层将 HTTP 请求的报文交给运输层处理
  + 运输层给 HTTP 请求报文添加一个 **TCP 首部（区分应用进程）**，使之成为 TCP 报文段。运输层将 TCP 报文段交给网络层处理
  + 网络层给 TCP 报文段添加一个 **IP 首部（使之可以在互联网上传输）**，使之成为 IP 数据报。网络层将 IP 数据报交付给数据链路层处理
  + 数据链路层给 IP 数据报添加一个**首部（让其能在一段链路上传输，能被相应主机接收）**和一个**尾部（让目的主机检查所接收到的帧是否有误码）**，使之成为**帧**。数据链路层将帧交给物理层
  + 物理层将帧看作比特流，以电信号的形式传输，接其发送到传输媒体

+ **服务器处理请求并返回一个HTTP响应报文** 

  浏览器收到 HTML 响应并渲染界面

## 2.6 OSI七层协议及每层作用

+ **物理层**  

  数据以比特流的形式在物理层传输，设备有中继器、集线器（作用：转发、放大信号）

+ **数据链路层** 

  负责将数据组帧，实现链路管理、流量控制和差错控制，设备有网桥、交换机（作用：根据 MAC 地址对帧进行过滤转发，可以隔绝冲突域，不能隔绝广播域）

  > **冲突域**：同一时间内只能有一台设备发送信息的范围
  > **广播域**：网络中能接收到任意设备发出的广播帧的设备的集合

  *停止-等待协议 SW，GBN后退N帧协议、SR选择重传协议*

+ **网络层**  

  提供主机和主机之间的逻辑通信，设备有路由器（作用：转发分组）

  ARP协议、OSPF/RIP路由寻址协议、DHCP协议、ICMP协议、IP协议

+ **传输层** 

  提供端到端的可靠报文传递，负责将数据传送至对应端口，提供进程和进程之间的逻辑通信

  *TCP，UDP协议*

+ **会话层**

  负责建立、管理、终止进程之间的会话

+ **表示层**

  对上层数据或者信息进行变换，以保证一个主机应用层信息可以被另一个主机应用层所理解，包括数据加密、格式转换等

+ **应用层**

  为操作系统或者网络应用程序提供访问网络的接口

  *HTTP，FTP，SMTP，DNS协议*

## 2.7 HTTP和HTTPS的区别

1. **端口号**：HTTP 端口号是 80；HTTPS 的端口号是 443

2. **建立连接**：HTTP三次握手后建立起无状态连接；HTTPS三次握手后还有SSL/TLS安全认证数据加密相关的握手操作

3. **数字证书**：HTTPS 需要CA申请数字证书证书，来保证服务器身份是可信的

4. **安全性**：HTTP以明文的形式传输；HTTPS认证后数据采用对称加密的方式传输

## 2.8 HTTPS安全连接建立

1. 客户端发起TLS连接请求
2. 服务器发送数字证书，其中包含数字签名，公钥，证书有效性等信息，同时保留唯一私钥
3. 客户端验证数字证书无误后，使用公钥加密对称密钥，发给服务器
4. 服务器使用私钥解密获得对称密钥
5. 进行数据传输，双方使用对称密钥对数据进行加密解密，服务器根据 Session ID 区分不同密钥

**为什么建立连接用非对称加密，而数据传输使用对称加密？**

1. 非对称加密相对而言安全性高，但是加解密效率低
2. HTTPS场景中只有服务器拥有私钥（公钥只能加密，私钥只能解密），因此客户端无法完成解密

## 2.9 常见状态码

+ **1xx：表示正在处理**

  + 100：服务器已收到客户端的请求头，并且客户端应继续发送请求体

+ **2xx：成功，表示请求已经正常处理**

  + 200：一切正常并返回数据
  + 204：请求正常处理，但是没有数据返回

+ **3xx：重定向，浏览器需要一些额外的操作才能完成请求**

  + 301：资源已永久移动到新位置，可能改变请求方式

  + 302：资源暂时移动到新网址，可能改变请求方式

  + 304：资源未改变，可以使用本地缓存

  + 307：资源临时移动到新位置，客户端应维持相同的请求方法

    > 302 类似 307，不过 302 会将请求方法改为 GET 再重定向至新 URL ，这可能会引起安全问题

  + 308：资源已永久移动到新位置，客户端应使用新的 URL 进行后续的请求，且应维持相同的请求方法

    > 301 与 308 的区别类似于 302 和 307 的区别

+ **4xx：客户端错误**

  + 400：URL请求错误
  + 401：身份认证不正确
  + **402：请求方法错误**
  + 403：请求正确，但服务器拒绝，一般是权限不足
  + 404：客户端申请访问的资源不存在

+ **5xx 服务器端错误**

  + 500：服务器在请求处理时内部出错
  + 503：服务器暂时无法处理请求，一般是达到连接上限或停机维护

## 2.9 cookie，session与token的区别

作用：为无状态的HTTP/HTTPS协议维护状态

**Cookie**：Cookie是**客户端**保持状态的方法

> 比如购物车如果是cookie保存，则下次使用购物车时，购物车内容就是从本地的cookie获得先前添加记录，而无需再去请求服务器获得，也由于存放在本地，因此容易被人破译其内容，相对没那么安全

+ Cookie就是存储由服务器发至客户端并由客户端保存的一段字符串
+ Cookie在客户端的保存形式有两种，一种是会话Cookie（浏览器关闭销毁），一种是持久Cookie（保存在磁盘）

**Session**：Session是**服务器**保持状态的方法

> 比如购物车如果是session保存，则下次使用购物车时，购物车内容是根据请求所携带的Session ID而获取对应Session得到的，也由于存放在服务器，因此相对来说难以被黑客攻破，相对安全一些

+ 第一次创建Session的时候，服务端会告诉客户端需要在Cookie里面记录一个Session lD，以后每次请求把这个会话ID发送到服务器
+ Session保存在服务器上，可保存在数据库、文件或内存中，每个用户有独立session
+ Session lD作为Session文件的Hash键，可以通过redis储存
+ 如果客户端的浏览器禁用了Cookie，会使用一种叫做URL重写的技术来进行会话跟踪，即每次HTTP交互，URL后面都会被附加上一个诸如 sid=xxxxx 这样的参数，服务端据此来识别用户

**Token**：服务器为客户端请求生成的令牌，客户端携带token访问服务器，服务器成功解析后返回对应数据


## 2.10 GET 和 POST 的区别

1. **语义**：get用于获取数据，幂等；post用于发送数据，不幂等

   > **一次请求和重复多次请求对资源的影响是一致的，就称该操作为幂等的**

2. **请求体**：get把请求的数据放在url上，以?分割URL和传输数据，参数之间以&相连，不安全；post把数据放在HTTP包内

3. **TCP数据包**：get产生一个TCP数据包，浏览器会把请求头与请求体一并发送，服务器响应200；post产生两个TCP数据包，浏览器先发送请求头，服务器响应 100 continue，随后再发送请求体，服务器响应200

4. **缓存**：get请求会被浏览器主动缓存，post不会，除非手动设置

# 4. Java

## 4.1 多线程

### 线程与进程

+ 程序由指令和数据组成，当一个程序被运行，从磁盘加载代码至内存时就开启了一个**进程**，一个进程包含多个线程，每个线程执行不同任务
+ 不同进程使用不同内存空间，在当前进程下的所有线程可以共享当前内存空间

### 并行与并发

+ 并行是同一时刻执行多个任务
+ 并发是同一时间段执行多个任务，时间片较小时看起来是“并行”

### 创建线程的方式

1. 继承Thread类，直接创建线程

   ```java
   public class Solution extends Thread{
           @Override
           public void run() {
               System.out.println("hello");
           }
       
   		public static void main(String[] args) {
   			Solution solution = new Solution();
   			solution.start();
   		}
   }
   ```

2. 实现Runable接口，作为Thread类构造参数创建线程

   ```java
   public class Solution implements Runnable{
   		@Override
   		public void run() {
   			System.out.println("hello");
   		}
   		
   		public static void main(String[] args) {
                // 传入实现 Runnable 接口的类实体构造线程
   			Thread thread = new Thread(new Solution());
   			thread.start();
   		}
   
   }
   
   ```

3. 实现Callable接口，相较于Runable多了异常向上抛出以及返回值

   ```java
   public class Solution implements Callable<String>{
   		@Override
   		public String call() throws Exception {
   			System.out.println("thread start...");
   			return "return_value";
   		}
   		
   		public static void main(String[] args) throws Exception{
                // 传入Callable接口实现类
   			FutureTask<String> futureTask = new FutureTask<>(new Solution());
   			Thread thread = new Thread(futureTask);
   			thread.start();
                // FutureTask实体类get方法获得call()返回值
   			System.out.println(futureTask.get());
   		}
   
   }
   ```

4. 通过线程池创建，传入Runable或Callable接口实现类都可以

   ```java
   public class Solution implements Runnable{
   		@Override
   		public void run() {
   			System.out.println("hello");
   		}
   		
   		public static void main(String[] args) {
                // 线程池大小为3
   			ExecutorService threadPool = Executors.newFixedThreadPool(3);
                // 往线程池中加入新线程，此时线程默认启动
   			threadPool.submit(new Solution());
                // 关闭线程池
   			threadPool.shutdown();
   		}
   
   }
   ```

### 调用run()和调用start()有什么区别

start方法是开启线程，run方法只是普通方法。直接调用run方法由当前线程负责执行，调用start方法则开辟新线程执行run()方法，当前线程并不会被阻塞【时间片轮询】

### wait()和sleep()相同与不同

**相同：**效果都是让线程暂时放弃CPU使用权，进入阻塞状态，都可以设置阻塞时间

**不同：**

1. wait()可以被notify()唤醒，sleep()必须等待足够时间

2. wait()是实体类方法且必须在对象锁里才能使用（synchronized代码块包裹）

   sleep()是Thread静态方法且无此限制

3. wait()方法执行后会释放对象锁，允许其他线程获得对象锁【我放弃CPU，其他线程可以用】

   若sleep()在synchronized代码块中执行，不会释放对象锁【我放弃CPU，其他线程也不准用】

2. 调用interrupt()方法中断线程

   + 打断正常线程，可以根据打断状态来标记是否退出线程

     ```java
     Thread current = Thread.currentThread();

     ```

### 说一下对volatile的理解

```java
static volatile int flag = false;
```

1. 保证线程间变量的可见性

### synchronized和Lock有什么区别

1. synchronized 是关键字，源码在jvm中，用C++实现

   ```java
   static final Object lock = new Object();
   // 加对象锁，未释放时同样的对象锁无法进入
   synchronized (lock) {
   	...
   }
   ```

   Lock 是接口【内置锁，如ReentrantLock，ReentrantReadWriteLock是其实现类】，源码由JDK提供，用java实现

   ```java
   Lock lock = new ReentrantLock();
   try {
       // 获取锁
       lock.lock();
       ...
   } finally {
       // 释放锁
       lock.unlock();
   }
   ```

2. 使用synchronized时，退出同步代码块时锁会自动释放

   使用 Lock 时需要手动调用 unlock() 方法释放锁

3. 二者都属于悲观锁，都具备基本互斥、同步、锁重入功能

   Lock提供了更多功能，如公平锁【先来先服务】

   ```java
   // 设置公平锁
   Lock lock = new ReentrantLock(true);
   ```

   可打断【等待中允许被interrupt】

   ```java
   // 允许打断的锁
   lock.lockInterruptibly();
   ```

   可超时【设置获得锁等待时间】

   ```java
   // 尝试获得锁，等待时间为5秒
   lock.tryLock(5, TimeUnit.SECONDS);
   ```

   多条件变量【允许为锁加条件，可以用于控制执行顺序，类似操作系统PV操作】

   ```java
   // 创建锁条件
   Condition c1 = lock.newCondition();
   // 获得锁
   lock.lock();
   // 条件阻塞
   c1.await();
   // 条件唤醒
   c1.signal();
   ```

2. 增加更多消费者
3. 使用RabbitMQ惰性队列扩大容积

### 4.3.3 ArrayList底层实现原理

+ 底层主要使用动态大小数组实现
+ 初始容量为0，当第一次添加数据后初始化容量为10
+ 每次扩容扩充为原容量的1.5倍，且每次都将旧数组内容拷贝至新数组
+ 添加元素时有以下操作：
  1. 确保数组已使用长度+1后足够存下下一个数据
  2. 如果+1后大于当前数组长度，调用grow方法扩容（原来1.5倍）
  3. 扩容完成后，将新元素放置于size位置上

### 4.3.6 HashMap实现原理

底层使用数组 + 链表/红黑树 实现，通过哈希值确定数组下标，如果计算所得下标一致则拼接至链表中。

JDK1.8之前采用拉链法（数组+链表），无红黑树，1.8后当链表长度大于8且数组长度大于64时会从链表转换为红黑树

### 4.3.7 HashMap添加数据

1. 判断键值对数组 table 是否为空或null，是则执行 resize() 进行扩容（初始化长度16）
2. 根据键值key计算hash得到数组索引
3. 如果 table[i]==null 成立，直接新建节点添加；如果不成立，则进行如下步骤：
   1. 判断 table[i] 首元素是否与 key 一致，相同直接覆盖 value
   2. 若 table[i] 为红黑树，则直接插入键值对，若已存在则直接覆盖value
   3. 若 table[i] 为链表，遍历链表，若已存在直接覆盖value，否则在尾部插入，此时若长度大于 8 且数组长度大于64，则将其转换为红黑树
4. 插入成功后，判断实际存在键值对数量是否超过最大容量threshold（数组长度 * 0.75），超过则进行扩容

### 4.3.8 HashMap扩容机制

+ 初始长度为16，后续达到阈值（数组长度 * 0.75 ）后触发扩容机制，扩充为原先容量的两倍
+ 扩容实则是创建一个新数组，将原先数据挪动至新数组中
  + 对于没有hash冲突的节点（即没有挂载链表），使用公式将其映射到新数组索引位置
  + 如果是链表，则对链表进行拆分（通过公式决定），该元素要么停留在原索引处，要么移动到原始位置 + 旧容量对应的位置上

## 5.5 装饰器

装饰器用于在不修改已有函数源代码的情况下，为已有函数增加额外功能或者修改函数的行为，其参数是被修饰的函数，返回值是一个新函数

```python
import time
def timing_decorator(func):
    # *args, **kwargs可以接收一切参数
    def wrapper(*args, **kwargs):
        start_time = time.time()
        # 执行原函数
        result = func(*args, **kwargs)
        end_time = time.time()
        print(f"Function {func.__name__} took {end_time - start_time:.4f} seconds")
        return result
    return wrapper

@timing_decorator
def slow_function():
    time.sleep(2)
    print("Function complete")

slow_function()
```

## 6.2 进程三大状态

+ 就绪态：进程等待分配资源
+ 运行态：进程占用资源，执行指令
+ 阻塞态：进程因为某些原因无法继续执行，等待后续唤醒

**状态转换关系**

+ **就绪态到运行态**：处理器空闲或当前运行的进程时间片用完，CPU通过调度算法选择一个处于就绪态的进程分配资源，使其进入运行态
+ **运行态到阻塞态**：当前运行的进程需要等待某个事件（如用户输入）完成时，进程会将自身切换为阻塞态，并将处理器资源释放
+ **阻塞态到就绪态**：进程阻塞等待的事件已完成，进程从阻塞态回到就绪态，等待处理器调度执行
+ **运行态到就绪态**：当运行的进程时间片耗尽或被高优先级进程抢占时，进程由运行态切换为就绪态

## 6.3 如何申请管理内存？

虚拟内存基于局部性原理，在内存中存放的是使用较为频繁的数据页。通过页表查询目标页是否在内存中，通过页表完成虚拟内存与物理内存的映射，不存在或失效时产生缺页中断，此时需要从外存找到对应页数据按照页面置换算法顶替内存中的页，并更新页表中页面存活状态

## 6.4 进程/线程/协程

> 基本定义：进程是资源分配的最小单位，线程是程序执行的最小单元，协程是用户态轻量级线程

**进程（Process）**

+ **定义**: 进程是操作系统分配资源的基本单位。每个进程都有独立的内存空间、文件描述符等系统资源。
+ **特点**: 进程之间是相互独立的，一个进程的崩溃不会影响其他进程。进程间通信通常比较复杂，需要使用进程间通信机制，如管道、消息队列、共享内存等。
+ **上下文切换**: 由于进程拥有独立的内存空间，操作系统在切换进程时需要保存和恢复大量的状态信息，开销较大。

**线程（Thread）**

+ **定义**: 线程是进程中的一个执行单元，同一个进程中的多个线程共享进程的内存空间和资源。
+ **特点**: 线程之间共享数据和资源，所以创建和切换线程的开销较小。线程之间的通信也比进程简单，可以直接访问相同的变量。但是，多个线程同时访问共享资源时需要小心处理同步问题，避免竞争条件。
+ **上下文切换**: 线程上下文切换的开销小于进程，因为线程共享内存空间。

**协程（Coroutine）**

+ **定义**: 协程是一种更轻量级的并发执行单元。协程不由操作系统内核管理，而是由程序自行调度。协程之间可以协作执行，即一个协程可以暂停自己并切换到另一个协程。
+ **特点**: 协程拥有自己的独立栈，但是它们共享同一个线程的上下文。协程切换的开销极小，因为无需操作系统介入。协程更适合处理大量I/O密集型任务，如网络请求，因其可以在等待I/O时让出控制权，而不是阻塞线程。
+ **上下文切换**: 协程的上下文切换开销最小，因为不涉及操作系统的线程调度。

## 6.5 进程通信方式

由于每个进程的用户空间是独立的，不能相互访问，这时就需要借助内核空间来实现进程间通信，因为每个进程都是共享一个内核空间的

+ **管道**
  
  管道是半双工通信，如果要双向通信，需要创建两个管道。通信的数据是无格式字节流且大小受限
  匿名管道只能用于存在父子关系的进程间通信，命名管道可以用于各个进程间通信
  不论哪种管道，进程写入的数据都是缓存在内核中，另一个进程读取数据时从内核获取，遵循先进先出原则。
  
+ **消息队列**

  消息队列实际上是保存在内核的消息链表，消息体可以是用户自定义的数据类型，每次数据的写入和读取都需要经过用户态和内核态之间的拷贝过程

+ **共享内存**：允许多个程序同时访问相同的内存空间，访问共享内存区域的进程需要注意同步问题

  共享内存可以解决消息队列通信中用户态与内核态之间数据拷贝过程带来的开销，它直接分配一个共享空间，每个进程都可以直接访问，就像访问进程自己的空间一样快捷方便，是进程间最快的通信方式，但是多进程竞争同个共享资源时会造成数据混乱（使用信号量机制）

+ **套接字Socket**：适用于不同机器之间的进程通信

## 6.6 死锁

死锁发生在多个线程或进程之间，因为每个线程或进程都在等待其他进程或线程释放资源，从而形成相互等待的循环

**产生死锁的必要条件**

1. **互斥条件**

   至少有一个资源只能同时被一个线程或进程调用

2. **请求与保持条件**

   线程或进程至少持有一个资源，并且在等待获取其他资源时不释放已持有的资源

3. **不可剥夺条件**

   资源只能在线程或进程主动释放情况下才能被其他线程或进程获取，外部不能强制剥夺资源

4. **循环等待条件**

   多个线程或进程之间形成循环，每个线程或进程都在等待下一个线程或进程持有的资源

**避免死锁的方式**

1. **破坏互斥条件**

   较难实现，因为互斥通常是不可避免的

2. **破坏请求与保持条件**

   要求线程请求资源时，必须先释放已经持有的资源，然后再请求所需资源。可以通过一次性获取所有需要资源来实现

3. **破坏不可剥夺条件**

   允许系统强制剥夺某个线程或进程持有的资源，可以通过超时等待或优先级机制实现

4. **破坏循环等待条件**

   为资源分配引入全局资源分配层次，使得线程或进程只能按照一定顺序请求资源，不会形成循环等待

经典问题：生产者消费者问题、读者写者问题、哲学家问题

## 6.7 同步与异步

+ 同步和异步是一种消息通信机制，同步就是调用者主动等待调用的结果，异步就是调用发出后没有直接返回结果（通过回调得到结果）
+ **同步阻塞**：问老板有没有书，站着等老板回答（等待过程就是阻塞）
+ **同步非阻塞**：问老板有没有书，等回复的过程中先去做别的事，时不时回来看一下找到没有（轮询）
+ **异步**：问老板有没有书，老板让你走，找到了主动联系你（回调）

## 6.8 堆和栈的区别

+ 申请方式不同
  + 栈由系统自动分配（连续存储空间）
  + 堆是自己申请和释放的（一个记录空闲内存地址的链表）
+ 申请大小限制不同
  + 栈顶和栈底是预设好的，栈向栈底扩展，大小固定
  + 堆向高地址扩展，是不连续的内存空间，大小可以灵活调整
+ 生命周期
  + 栈上的数据对象的生命周期与其所在函数执行周期相对应。一旦函数执行完毕，栈上的局部变量会被自动销毁，内存会被回收
  + 堆上的数据对象生命周期可以比函数执行周期更长，可以通过显式释放内存控制其生命周期。如果没有手动释放堆上的内存，可能会导致内存泄漏。
+ 访问速度不同
  + 操作系统在底层对栈提供支持，分配有专门的寄存器且有专门指令，因此效率较高
  + 堆在分配堆内存时需要按照一定算法寻找合适大小内存，并且堆的内容需要两次访问，一次访问指针，第二次根据指针保存的地址访问内存，因此比较慢

## 6.9 原码反码补码

+ 正数三个码均相等，即符号位+绝对值

+ 负数反码 = 符号位不变，其余按位取反

  负数补码 = 原码取反码 + 1

$$
(-x)_补 = x_补按位取反【包括符号】 + 1 \\
(x-y)_补 = x_补 - y_补 = x_补+(-y)_补
$$



# 7. 测试

## 7.1 CI/CD是什么

CI/CD 是指持续集成（Continuous Integration）和持续交付（Continuous Delivery）与持续部署（Continuous Deployment）的结合，它是一个软件开发流程，旨在通过自动化构建、测试发布和部署的方式来加快应用程序的开发和发布速度，同时保证应用程序的质量和稳定性。

+ 持续集成是指将团队中开发人员的代码频繁集成到主干代码库；
+ 持续交付是在完成持续集成后，自动部署代码到测试和预发布环境，以便进一步测试评估
+ 持续部署则是在完成持续交付后，自动将应用程序部署到生产环境

## 7.2 什么是测试左/右移

测试左移侧重于早期测试和质量保证（如测试用例设计，代码评审），测试右移侧重于真实环境下的测试和性能优化（如监测性能指标，日志分析），二者都是为了更好的发现和解决缺陷，提高软件质量。

## 7.4 接口测试是怎么做的

1. 阅读接口文档，关注点在于请求方式、请求参数、返回值，业务逻辑这几个方面
2. **请求参数**覆盖必填参数，必填+非必填组合、多参少参无参、参数类型不匹配的情况
3. 对不同情况下的接口返回值进行断言，观察是否符合预期
4. 业务逻辑指多个接口请求之间是否存在依赖关系
5. 在测试前需要保证测试环境没有出现问题，如Web服务器是否顺利开启，数据库服务是否正常运行等

## 7.5 接口的自动化断言怎么做

+ 返回结果为JSON，采用键值对断言，关键字对应val与预期一致则断言通过
+ 涉及数据库查询或修改的接口，需要进行SQL校验，看数据库中数据是否符合预期
+ 接口返回文本，可以对文本中具有分辨度的内容断言
+ 对请求头的响应状态码断言

## 7.6 性能测试指标

+ 吞吐量：系统在单位时间内能够处理的请求数量，每秒完成事务数TPS
+ CPU利用率/内存利用率：系统运行过程中CPU和内存的占用情况
+ 异常率：测试过程中出现错误的请求占总体请求的百分比
+ 响应时间：服务器处理请求的时间
+ 网络延迟：请求和响应之间的时间差

## 7.7 遇到线上严重问题如何解决

1. 想办法及时止损，简述故障基本情况交给开发处理，后续通过测试后完成更新
   1. 故障基本情况简述
   2. 故障时间：发生时间点、发现的时间点
   3. 故障影响：业务范围
2. 故障修复后完成复盘
   1. 故障是由于哪个环节出现问题导致（需求阶段、开发阶段、测试阶段），具体到指定环节的责任人
   2. 为什么发生？为什么测试阶段没有发现？系统容错处理？能不能更早发现？解决过程能不能更快？怎么防止下次发生类似事情

## 7.10 逻辑测试覆盖中，覆盖程度由弱到强顺序是什么？

> 覆盖是测试人员之间常用的交流语言，也即测试到的地方称为覆盖

1. 语句覆盖 ：可执行语句至少被执行一次

   > 每行代码都被执行

2. 判定覆盖：每个（整体）判断的取真/取假分支至少经历一次

   > 关注分支结构（if...else等）的 True 和 False 分支是否能被正确覆盖

3. 条件覆盖：每个条件取值至少满足一次（不关心整体及排列组合）

   > 在整个测试过程中，逻辑判断子条件取值（True 或 False）至少一次，但是不关心取值的排列组合

   ```java
   public void foo (int a, int b, int x) {
       if(a>1 && b==0) {
           x = x/a;
       }
       if (a==2 || x>1) {
           x = x+1;
       }
   }
   ```
   
   第一个判断的所有条件的可能取值情况是 a>1 或 a≤1 ，b=0 或 b≠0 ；第二个判断条件可能的取值情况为 a=2 或 a≠2 ，x>1 或 x≤1
   
   ```java
   // 以下测试用例即可满足所有条件覆盖（所有条件值都被取到）
   a=1，b=0，x=3 
   a=2，b=1，x=1 
   ```
   
   然而在这种情况下第二个判断条件却无法取值为 False，因此满足条件覆盖不一定满足判定覆盖
   
4. 判定/条件覆盖：判定覆盖和条件覆盖都要满足

5. 条件组合覆盖：每个条件的所有可能都至少出现一次，并且判定结果至少出现一次

   > 关注关注分支结构（if...else等）中所有单个子条件的 True 和 False 组合情况是否能被正确覆盖

   ```java
   public static void main(String[] args) {
   		Scanner sc=new Scanner(System.in);
   		int x=sc.nextInt();
   		int y=sc.nextInt();
   		int num=sc.nextInt();
   		if(x>0 && y>0) {
   			System.out.println("Hello1");
   		}else{
   			System.out.println("Hello2");
   		}
   		if(num < 0) {
   			System.out.println("Hello3");
   		}
   		System.out.println("Hello4");
   	}
   ```

   + 第一个 if 语句（x>0 && y>0）定义为 test1
   + 第二个 if 语句（num < 0）定义为 test2
   + x>0 定义为 p1；y>0 定义为 p2；num<0 定义为 p3

   以下测试用例满足条件覆盖

   | 测试数据           | p1   | p2   | p3   | test1 | test2 |
   | ------------------ | ---- | ---- | ---- | ----- | ----- |
   | {x=1,y=1,num=2}    | T    | T    | F    | T     | F     |
   | {x=-1,y=-1,num=-2} | F    | F    | T    | F     | T     |
   | {x=1,y=-1,num=2}   | T    | F    | F    | F     | F     |
   | {x=-1,y=1,num=-2}  | F    | T    | T    | F     | T     |

   满足条件组合覆盖可能不会覆盖所有的路径，如 test1=True 与 test2=True 的情况此处未被覆盖

6. 路径覆盖：执行所有可能的执行路径

   > 将程序执行抽象为流程图后所有路径进行覆盖

## 7.12 缺陷等级定义规范

1. **致命缺陷**：软件核心功能完全失效

   > 例如微信通讯，支付，朋友圈等服务完全崩溃

2. **严重缺陷**：软件出现严重错误，影响部分核心功能

   > 例如微信朋友圈功能异常，但是通讯、支付等服务正常

3. **一般缺陷**：软件次要功能受到影响，但不影响核心业务

   > 例如购物平台中商品详情页加载缓慢，但不影响用户购买

4. **轻微缺陷**：不影响功能实现，只对用户体验有轻微影响

   > 例如博客平台上，部分文章字体显示与设计规范不符

## 7.13 W模型和V模型

> 二者都是在软件开发过程中用于指导测试活动的模型

<img src="https://z-cloud-pic-1313046262.cos.ap-guangzhou.myqcloud.com/img/202408191422078.png" alt="image-20240819142239994" style="zoom:67%;" />

以开发手机照相功能为例，W模型的各个阶段任务如下：

1. **用户需求**：用户提出对于照相功能的期待，明确需要实现的功能和特性，比如自动对焦，高清拍摄等

   验证：审查需求文档，确保用户需求是可被理解且没有遗漏的

2. **需求分析**：详细分析用户需求，编写需求规格说明书，业务实现流程，如自动对焦的具体指标，高清的具体定义

   验证：评审需求规格说明书，确保需求，业务流程被正确理解且是可测试的

3. **概要设计**：设计照相功能的整体架构，定义各模块的划分以及模块间的接口，用户操作实现逻辑等

   验证：审查概要设计文档，确保设计可以满足需求文档，系统架构合理，模块划分清晰

4. **详细设计**：详细描述每个模块的内部设计和实现细节。比如调节光圈大小需要分为几个单元，每个单元需要完成的任务是什么

   验证：对详细设计文档进行评审，确保设计详尽，每个模块的接口和功能明确

5. **编码**：根据设计文档编写代码实现系统功能

6. **模块**：通过白盒方法测试编写的各个模块，是**单元测试**

7. **集成**：通过调用顺序集成各个模块实现一个完整业务功能，通过**集成测试**验证功能是否实现

8. **实施**：集成阶段已完成基本功能实现与联调，实施阶段进行更大规模的集成，确保整体功能可以顺利实现，同时对用户界面、易用性，性能、兼容性、安全性等方面进行测试，此处是**系统测试**

9. **交付**：此处已基本完成软件开发测试工作，最后对用户体验进行测试，以确保成品确实符合用户预期，即**验收测试**

## 7.14 扣款成功而订单生成失败

扣款成功 --> 服务器接收扣款成功响应 --> 服务器发送订单生成请求 --> 数据库生成订单 --> 服务器接收数据库响应 --> 用户接收响应

2. 订单生成请求未能发送成功
3. 数据库生成订单操作失败

5. 服务器收到数据库响应后处理出错

# 8. 非专业知识面试

## 8.1 为什么选择测开而不是开发

我自己在学习过程中累积了相当代码基础，但也发现自己对于追求技术深度并不是特别有热情，反倒对具体的业务实现所涉及的知识以及错误出现的原因会更有兴趣，后来我在一些技术博客上了解到测试开发这个岗位，它相对于纯测试来说还额外需要一定的开发能力，更加注重知识的广度，核心是做软件质量保障，开发测试工具或者测试平台去提升测试效率，这非常符合我的定位。加之后续针对测试相关理论知识以及去企业进行实习后，对于这一岗位的工作内容依旧怀有热情，所以最后选择往测试开发的路线发展。

## 8.2 遇到开发不认可的BUG如何处理

1. 排除非BUG的可能性（自己检查一遍）

   > 再次对照需求文档和技术方案，排查测试环境和数据等，确保BUG并非由测试人员疏忽所导致

2. 详细描述BUG

   > 确保BUG复现步骤、错误现象以及预期结果等信息描述足够清除，确保截图、日志等证据足够有说服力

3. 强调BUG重要性

   > 说明此BUG对系统功能、用户体验或者业务流程的影响，解释清楚为什么这个BUG应该被认可和修复

4. 进行有效沟通

   > 与开发人员进行积极有效沟通，尊重开发人员的专业知识和经验，试图理解他们的立场观点，共同探讨BUG出现的原因，解决方案以及优先级，以达成一致

## 8.3 令人印象深刻的BUG

我在实习时参与测试的一项任务是创建某个关键物料，然后将其推送至MDM平台，相关人员可以从MDM平台获取到自己所需要的物料。当时我按照说明将物料创建好推送后，其他需要使用此物料的人却反馈说在平台中没有查询到此物料，而我通过前后端抓包分析后发现一切正常，因此询问了带我的测试组长，最后发现是因为测试环境不一致导致的，切换环境后操作结果符合预期。

因为自己先前在本地练习的项目没有切换环境这一要求，加之也是我在实习时遇到的第一次BUG，所以印象还是比较深刻的，经过这次教训后，每次在进行测试前我都会先对一下测试环境，避免出现类似的低级错误。

## 8.4 公司的测试流程

我所了解的流程大概是这样，首先在需求阶段，测试、产品经理和开发人员相关负责人会一起参与需求评审和风险评估，接着拟出开发计划方案，测试计划方案等，确保项目按计划稳步前进，像我实习这边是采用敏捷开发的模式，几乎每周都会迭代出一个新的版本。

开发人员完成相关模块任务后会通知测试进行相应测试，测试人员则根据具体业务设计测试用例和脚本完成测试工作，这一过程中发现BUG后提交至缺陷管理平台，与开发人员积极沟通合作，直至测试覆盖率达到要求，整体达到准出标准。

反复迭代多次后最终得到实际产品，进行验收测试，先是内部人员完成α测试，后续扩大范围完成β测试，项目上线投入使用，跟踪用户反馈情况，以便后续进一步完善产品。

## 8.5 测试到什么程度可以上线

一般情况下，测试用例执行通过率达100%就可以上线，如果情况比较紧急，开发来不及解决所有BUG，则可以优先评估P0、P1级等严重影响用户体验的BUG，全部解决都再考虑产品是否通过，当然具体的标准需要经过实际测试团队评估才能确定。

## 8.6 自动化测试缺点

+ 自动化测试虽然能提高测试效率，但是一般来说手工测试更加灵活，测试覆盖率会更高
+ 自动化测试需要以手工测试为基础，通常得先经过手工测试才能进行自动化测试
+ 手工测试发现的BUG比自动化测试发现的BUG多得多
+ 自动化测试维护成本比较高，同时也有执行失败的几率
